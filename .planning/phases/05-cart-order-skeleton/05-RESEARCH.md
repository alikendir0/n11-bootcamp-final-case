# Phase 5: Cart & Order Skeleton ŌĆö Research

**Researched:** 2026-04-30
**Domain:** Spring Boot 3.5.14 / Java 21 microservices, choreography saga (RabbitMQ + transactional outbox), Postgres 16 schema-per-service, Stripe-style HTTP idempotency, ArchUnit architecture-test enforcement, Testcontainers + Awaitility multi-service E2E
**Confidence:** HIGH (CONTEXT.md pre-locks 11 decisions; canonical patterns from Phase 3 + Phase 4 already in repo; all architectural choices verified against official docs and existing source)

---

## Summary

Phase 5 stands up three new Spring Boot services (`cart-service`, `order-service`, `payment-service` skeleton) plus a new shared Gradle library (`common-outbox`) that lifts the outbox-poller pattern out of `inventory-service` + `identity-service` into a single home. The saga skeleton runs end-to-end (`order.created` ŌåÆ `stock.reserved` ŌåÆ `payment.completed` ŌåÆ `order.confirmed`) with **all four compensation paths wired** even though only the happy path actually fires in Phase 5 testing ŌĆö Phase 6 (real Iyzico) and Phase 5 user-cancel both need the consumer plumbing already in place.

The five biggest implementation risks are (1) **dual-writes** (kept off by the outbox pattern; **non-negotiable**), (2) **non-idempotent consumers** (CLAUDE.md Rule #3 ŌĆö every consumer must use `processed_events` inbox + redelivery test), (3) **silent AMQP regressions** (D-10 ArchUnit gate enforces `AcknowledgeMode.AUTO` invariant from the Phase 4 Plan 04-03 retrospective), (4) **leaky transactions** (sync REST calls to product-/identity-service must happen BEFORE `@Transactional` opens, never inside), and (5) **missing message_id property** (the `StatefulRetryInterceptor.messageKeyGenerator` rejects messages without `MessageProperties.setMessageId(eventId)` ŌĆö D-09's shared `OutboxMessagePostProcessor` makes this invariant uniform across all 4 outbox publishers).

**Primary recommendation:** Plan in roughly 4ŌĆō5 plans across 4 waves: (W0) `common-outbox` shared module + identity- and inventory-service migrations + D-10 ArchUnit gate, (W1) `cart-service` end-to-end (REST + repository + cart.q.order-confirmed consumer), (W2) `order-service` end-to-end (REST + Idempotency-Key + outbox publisher + 4 saga consumers + state machine), (W3) `payment-service` skeleton + `infra-tests` happy-path E2E + inventory-service compensation consumer wiring, (W4) gateway routes + Springdoc aggregator + Phase 5 smoke runbook.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-01 ŌĆö Cart Line-Item Snapshot Strategy:** Snapshot `unitPriceSnapshot` + `nameSnapshot` + `image_url_snapshot` + `added_at` at add-time AND on quantity-update (cart-service synchronously calls `GET /products/{id}` and stores those fields on the `cart_items` row). `GET /cart` returns snapshotted fields directly (no per-read product-service round-trip). At `POST /orders`, order-service re-fetches `GET /products/by-ids` (or per-product loop if no batch endpoint exists yet) to re-validate prices; on any drift (strict equality for v1) return `409 Conflict` with `{"updatedItems": [...]}` so the frontend can re-render and re-confirm.

**D-02 ŌĆö Cart Line-Item Identifier:** `{id}` in `/cart/items/{id}` is `productId` (UUID). Composite primary key on `cart_items(user_id, product_id)`; productId is unique-per-cart by construction. `POST /cart/items {productId, qty}` is an UPSERT (`INSERT ... ON CONFLICT (user_id, product_id) DO UPDATE SET qty = cart_items.qty + EXCLUDED.qty, ...`).

**D-03 ŌĆö Order-Creation Flow (sync vs async):** Pure choreography. `POST /orders` returns **`202 Accepted` with `{orderId, status: "PENDING"}` immediately**. Order-service writes `orders` + `order_items` + `order_shipping_addresses` + `outbox(order.created)` in **ONE `@Transactional` boundary**. Outbox poller publishes `order.created`. Frontend polls `GET /orders/{id}`. NO synchronous wait for the saga.

**D-04 ŌĆö Address Snapshot at Order Creation:** Separate `order_shipping_addresses` table in the `orders` schema, one row per order, FK to `orders(id)`. Mirrors Phase 3 D-09's `addresses` table fields (`recipient_name`, `phone`, `il`, `ilce`, `mahalle`, `street_line`, `postal_code`, `title`). `order_id` is the PK (1:1 with order); no soft-delete (orders are immutable after creation). Order-service calls `GET /api/v1/identity/addresses/{addressId}` (gateway-routed, JWT-forwarded as `X-User-Id`); identity-service must enforce ownership (return 404 if address not owned by `X-User-Id`).

**D-05 ŌĆö Order Idempotency on `POST /orders`:** Required `Idempotency-Key` HTTP header on `POST /orders` (UUID generated by frontend). Order-service maintains an `order_idempotency_keys(idempotency_key UUID PK, user_id UUID, order_id UUID, created_at TIMESTAMPTZ)` table; UNIQUE constraint on `(idempotency_key, user_id)`. Repeat key by SAME user ŌåÆ return existing `orderId` with HTTP **200** (not 201). Same key by DIFFERENT user ŌåÆ 400. Missing header ŌåÆ 400 RFC-7807. **CD-NOTE:** No TTL cleanup job in v1 ŌĆö rows kept forever (~30 grader orders is nothing).

**D-06 ŌĆö Phase 5 Payment-Service Scope:** Ship a ~200-LOC payment-service skeleton ŌĆö Spring Boot app, port 8086, schema `payment`, Eureka-registered, **NOT routed through gateway in Phase 5** (no public REST yet ŌĆö Phase 6 adds `/payments/checkout` + the gateway route). Single consumer: `StockReservedConsumer` listening on `payment.q.stock-reserved`. On consume: idempotency check via `processed_events`, write a `payments` row with `status='COMPLETED'`, write outbox row with `eventType='payment.completed'` (after `mock.payment.delay-ms: 100`, configurable; 0 disables for tests). NO `PaymentProvider` port (Phase 6 owns it).

**D-07 ŌĆö Cart Clearing on `order.confirmed`:** Clear ENTIRE cart on `order.confirmed` consumer in cart-service ŌĆö `DELETE FROM cart_items WHERE user_id = ?` (the `carts` row stays). v1 has no partial checkout. Edge case (user adds item between `POST /orders` and `order.confirmed` arriving ŌĆö gets cleared too) is acceptable for v1 sub-second windows.

**D-08 ŌĆö Order Status Enum + Saga Ōåö Display-Label Mapping:** Backend canonical `OrderStatus.java` = exactly 6 states + cancellation tail: `PENDING ŌåÆ STOCK_RESERVED ŌåÆ PAID ŌåÆ CONFIRMED`, plus `STOCK_FAILED ŌåÆ CANCELLED`, `PAYMENT_FAILED ŌåÆ CANCELLED`, and `USER_CANCELLED` direct from PENDING/STOCK_RESERVED. Add `cancel_reason VARCHAR(64)` column for `OUT_OF_STOCK | PAYMENT_DECLINED | USER_CANCELLED | PAYMENT_TIMEOUT`. **Turkish display timeline ("Sipari┼¤ Al─▒nd─▒ ŌåÆ Haz─▒rlan─▒yor ŌåÆ Kargoya Verildi ŌåÆ Teslim Edildi") is FRONTEND-ONLY** ŌĆö Phase 10 maps {`PENDING`,`STOCK_RESERVED`} ŌåÆ "Sipari┼¤ Al─▒nd─▒"; {`PAID`,`CONFIRMED`} ŌåÆ "Haz─▒rlan─▒yor"; "Kargoya Verildi" + "Teslim Edildi" are static placeholder steps (greyed). Backend exposes canonical enum + `cancelReason` + `updatedAt`.

**D-09 ŌĆö `common-outbox` Shared Module Extraction:** Create new `common-outbox` Gradle library module exporting (1) `OutboxEvent` JPA entity, (2) `AbstractOutboxPoller` with `@Scheduled(fixedDelay=5000)` driver (concrete subclass per service supplies its own `OutboxRepository`), (3) **critical** `OutboxMessagePostProcessor` reading the envelope's `eventId` from the outbox-row payload JSON and setting `MessageProperties.setMessageId(eventId)` on every AMQP publish. Migrate existing identity-service + inventory-service pollers to consume the shared module; their tests run unchanged.

**D-10 ŌĆö 999.2 Architecture-Test Enforcement:** Add an ArchUnit test in `infra-tests/` that fails the build if any class annotated with `@RabbitListener` lives in a context whose factory uses `AcknowledgeMode.MANUAL` AND the listener method does not have a `Channel` parameter. Phase 5 introduces 3+ new consumers; without this gate the AMQP-ack regression fixed in Plan 04-03 commit `2b61689` can silently come back.

**D-11 ŌĆö Cart-Service Schema (One-User-One-Cart):** One cart per user, lazy-create on first `POST /cart/items`. `carts(user_id UUID PK, updated_at TIMESTAMPTZ)`. `cart_items(user_id, product_id, qty, unit_price_snapshot, name_snapshot, image_url_snapshot, added_at)` with composite PK `(user_id, product_id)`. `userId` from gateway-injected `X-User-Id` header (no JWT decoding in cart-service). `GET /cart` lazy-creates an empty `carts` row.

**`AbstractOutboxPoller`:**
```java
package com.n11.outbox;

import com.fasterxml.jackson.databind.JsonNode;
import com.fasterxml.jackson.databind.ObjectMapper;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.amqp.core.MessagePostProcessor;
import org.springframework.amqp.core.MessageProperties;
import org.springframework.amqp.rabbit.core.RabbitTemplate;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.transaction.annotation.Transactional;

import java.time.Instant;
import java.util.List;

public abstract class AbstractOutboxPoller {

    private static final Logger LOG = LoggerFactory.getLogger(AbstractOutboxPoller.class);
    private static final int DEFAULT_BATCH_SIZE = 100;

    protected final OutboxRepository outboxRepository;
    protected final RabbitTemplate rabbitTemplate;
    protected final ObjectMapper objectMapper;

    protected AbstractOutboxPoller(OutboxRepository outboxRepository,
                                   RabbitTemplate rabbitTemplate,
                                   ObjectMapper objectMapper) {
        this.outboxRepository = outboxRepository;
        this.rabbitTemplate = rabbitTemplate;
        this.objectMapper = objectMapper;
    }

    @Scheduled(fixedDelay = 5000)
    @Transactional
    public void poll() {
        List<OutboxEvent> unsent = outboxRepository.findUnsentBatch(DEFAULT_BATCH_SIZE);
        if (unsent.isEmpty()) return;

        for (OutboxEvent event : unsent) {
            String exchange = event.getAggregate() + ".tx";
            rabbitTemplate.convertAndSend(
                exchange,
                event.getEventType(),
                event.getPayload(),
                outboxMessagePostProcessor()
            );
            event.markSent(Instant.now());
            outboxRepository.save(event);
        }
    }

    /** Sets MessageProperties.setMessageId(eventId) ŌĆö extracts envelope.eventId from payload JSON. */
    protected MessagePostProcessor outboxMessagePostProcessor() {
        return message -> {
            try {
                JsonNode envelope = objectMapper.readTree(message.getBody());
                String eventId = envelope.path("eventId").asText(null);
                if (eventId == null || eventId.isBlank()) {
                    throw new IllegalStateException("Outbox payload missing envelope.eventId ŌĆö cannot set messageId");
                }
                MessageProperties props = message.getMessageProperties();
                props.setMessageId(eventId);
                props.setContentType("application/json");
                // Phase 5 ALSO sets correlation_id property from envelope (api-contracts.md ┬¦5):
                String correlationId = envelope.path("correlationId").asText(null);
                if (correlationId != null && !correlationId.isBlank()) {
                    props.setHeader("X-Correlation-Id", correlationId);
                    props.setCorrelationId(correlationId);
                }
            } catch (Exception e) {
                throw new IllegalStateException("Failed to inject messageId/correlationId from outbox envelope", e);
            }
            return message;
        };
    }
}
```

### Pattern 2: Cart UPSERT (D-02 / Postgres native + Hibernate composite-key entity)
**What:** Composite-PK JPA entity backed by a native UPSERT query; Spring Data JPA passes the SQL through verbatim.
**When to use:** Cart `POST /cart/items {productId, qty}` (server-driven add-or-increment).

**`CartItem` JPA shape with `@IdClass` (preferred over `@EmbeddedId` because `@IdClass` keeps property accessors flat ŌĆö `cartItem.getUserId()` not `cartItem.getId().getUserId()`):**

```java

    @Column(name = "added_at", nullable = false) private Instant addedAt;
```

**UPSERT via native query (in `CartItemRepository`):**

```java
public interface CartItemRepository extends JpaRepository<CartItem, CartItemId> {

    @Modifying
    @Query(value = """
        INSERT INTO cart.cart_items
            (user_id, product_id, qty, unit_price_snapshot, name_snapshot, image_url_snapshot, added_at, updated_at)
        VALUES
            (:userId, :productId, :qty, :unitPriceSnapshot, :nameSnapshot, :imageUrlSnapshot, now(), now())
        ON CONFLICT (user_id, product_id) DO UPDATE SET
            qty                  = cart_items.qty + EXCLUDED.qty,
            unit_price_snapshot  = EXCLUDED.unit_price_snapshot,
            name_snapshot        = EXCLUDED.name_snapshot,
            image_url_snapshot   = EXCLUDED.image_url_snapshot,
            updated_at           = now()
        """,
        nativeQuery = true)
    int upsert(@Param("userId") UUID userId,
               @Param("productId") UUID productId,
               @Param("qty") int qty,
               @Param("unitPriceSnapshot") BigDecimal unitPriceSnapshot,
               @Param("nameSnapshot") String nameSnapshot,
               @Param("imageUrlSnapshot") String imageUrlSnapshot);
}
```

[CITED: postgresql.org/docs/16/sql-insert.html ŌĆö `ON CONFLICT (cols) DO UPDATE SET col = EXCLUDED.col`] ŌĆö Postgres 16 syntax verified.

**Rationale for native-query + `@IdClass`:**
- Hibernate 6.5+ has portable `INSERT ... ON CONFLICT DO UPDATE` JPQL but it doesn't compose cleanly with the `qty = cart_items.qty + EXCLUDED.qty` increment expression on a composite key.
- Spring Data JPA's `save()` does a SELECT-then-INSERT-or-UPDATE round-trip, which races under concurrent adds (the chat assistant's `add_to_cart` tool and the web "Sepete Ekle" button can fire simultaneously for the same user).
- Native UPSERT is atomic at the DB level ŌĆö no race.
- After the UPSERT, callers that need the updated row should `entityManager.clear()` then `findById(new CartItemId(userId, productId))` to read it back (the `@Modifying` query bypasses Hibernate's first-level cache).

[CITED: w3tutorials.net "How to Upsert Entities in Spring Data JPA/Hibernate with Postgres"] ŌĆö composite unique constraint pattern.

### Pattern 3: Idempotency-Key on `POST /orders` (D-05 / Stripe-style)
**What:** Required `Idempotency-Key` HTTP header; UUID generated by frontend; server stores `(idempotency_key, user_id, order_id)`; repeats return existing orderId with HTTP 200.
**When to use:** EVERY mutating endpoint where the client may retry (here: only `POST /orders` in Phase 5 ŌĆö `POST /cart/items` doesn't need it because UPSERT is naturally idempotent for `(user_id, product_id)`).

**Storage table (in `orders` schema):**

```sql
CREATE TABLE order_idempotency_keys (
    idempotency_key UUID         NOT NULL,
    user_id         UUID         NOT NULL,
    order_id        UUID         NOT NULL REFERENCES orders(id),
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
    PRIMARY KEY (idempotency_key, user_id)
);
```

[CITED: brandur.org/idempotency-keys ŌĆö Stripe-style schema] simplified: store `order_id` not full response (microservices reconstruct the response from the orders table on retry).

**Controller signature:**

```java

```

[CITED: docs.stripe.com/api/idempotent_requests ŌĆö Stripe Idempotency-Key semantics] ŌĆö repeats return same response.

**Service flow (no request_body hash in v1 ŌĆö recommended simplification):**

```
1. SELECT order_id FROM order_idempotency_keys WHERE idempotency_key=? AND user_id=?
   ŌĆó Found  ŌåÆ SELECT order from orders by that order_id; return as 200 with existing orderId  (REPLAY)
   ŌĆó Conflict (key exists for DIFFERENT user) ŌåÆ 400 (suspicious; do NOT leak existence to caller)
   ŌĆó Not found ŌåÆ continue to step 2
2. Re-validate prices vs cart (D-01 strict equality) ŌĆö happens BEFORE @Transactional opens
   ŌĆó Drift ŌåÆ 409 with {"updatedItems": [...]} BEFORE writing anything
3. BEGIN @Transactional:
     INSERT orders, order_items, order_shipping_addresses
     INSERT order_idempotency_keys (idempotency_key, user_id, order_id, created_at=now())
     INSERT outbox(order.created envelope)
   COMMIT
4. Return 202 with new orderId  (FIRST CALL)
```

**Race condition note:** if two simultaneous requests with the same `(idempotency_key, user_id)` arrive, the PRIMARY KEY constraint causes the second to throw `DataIntegrityViolationException` at commit time. The service catches this exception and re-runs step 1 to fetch the order_id of the winner ŌĆö caller sees the same response.

**TTL note (CD-NOTE in CONTEXT.md):** v1 keeps idempotency rows forever. Stripe pruning at 24h is a v2 nicety; not in scope.

### Pattern 4: ArchUnit Ack-Mode Gate (D-10)
**What:** A test in `infra-tests/` that fails the build if any class annotated with `@RabbitListener` is in a context whose container factory uses `AcknowledgeMode.MANUAL` AND the listener method does not have a `Channel` parameter.
**When to use:** Once, in Phase 5; runs every CI build.

**Approach: ArchUnit can scan classpath for `@RabbitListener`-annotated METHODS and inspect parameter types.** ArchUnit cannot inspect the bean wiring (the `containerFactory` reference is a string), so the test focuses on what it CAN see: the listener's method signature shape.

**Concrete rule shape:**

```java
package com.n11.infratests.arch;

import com.tngtech.archunit.core.domain.JavaClasses;
import com.tngtech.archunit.core.importer.ClassFileImporter;
import com.tngtech.archunit.core.importer.ImportOption;
import com.tngtech.archunit.lang.ArchCondition;
import com.tngtech.archunit.lang.ArchRule;
import com.tngtech.archunit.lang.SimpleConditionEvent;
import com.rabbitmq.client.Channel;
import org.junit.jupiter.api.Test;
import org.springframework.amqp.core.Message;
import org.springframework.amqp.rabbit.annotation.RabbitListener;

import static com.tngtech.archunit.lang.syntax.ArchRuleDefinition.methods;

class AmqpAckModeArchTest {

    private final JavaClasses CLASSES = new ClassFileImporter()
        .withImportOption(ImportOption.Predefined.DO_NOT_INCLUDE_TESTS)
        .importPackages("com.n11");

    /**
     * D-10: Every @RabbitListener method MUST take an {@code org.springframework.amqp.core.Message}
     * parameter (the canonical "AUTO ack mode + RejectAndDontRequeueRecoverer" shape) AND MUST NOT
     * declare a {@code com.rabbitmq.client.Channel} parameter (which is the signature for the
     * MANUAL ack mode that re-introduces the Plan 04-03 regression silently).
     *
     * <p>If a future consumer legitimately needs MANUAL ack with Channel, the test must be updated
     * deliberately ŌĆö that's the point of the gate.
     */
    @Test
    void rabbitListenersUseAutoAckShape() {
        ArchRule rule = methods()
            .that().areAnnotatedWith(RabbitListener.class)
            .should(notDeclareChannelParameter())
            .andShould(declareMessageParameter())
            .as("All @RabbitListener methods must use AUTO-ack shape: Message parameter, no Channel parameter (D-10 / Plan 04-03 regression gate)");

        rule.check(CLASSES);
    }

    private static ArchCondition<com.tngtech.archunit.core.domain.JavaMethod> notDeclareChannelParameter() {
        return new ArchCondition<>("not declare a com.rabbitmq.client.Channel parameter") {
            @Override
            public void check(com.tngtech.archunit.core.domain.JavaMethod method, com.tngtech.archunit.lang.ConditionEvents events) {
                boolean hasChannel = method.getRawParameterTypes().stream()
                    .anyMatch(t -> t.isAssignableTo(Channel.class));
                if (hasChannel) {
                    events.add(SimpleConditionEvent.violated(method,
                        method.getFullName() + " has Channel parameter ŌĆö MANUAL ack mode regression (D-10)"));
                }
            }
        };
    }

    private static ArchCondition<com.tngtech.archunit.core.domain.JavaMethod> declareMessageParameter() {
        return new ArchCondition<>("declare a Spring AMQP Message parameter") {
            @Override
            public void check(com.tngtech.archunit.core.domain.JavaMethod method, com.tngtech.archunit.lang.ConditionEvents events) {
                boolean hasMessage = method.getRawParameterTypes().stream()
                    .anyMatch(t -> t.isAssignableTo(Message.class));
                if (!hasMessage) {
                    events.add(SimpleConditionEvent.violated(method,
                        method.getFullName() + " has no Message parameter ŌĆö non-canonical listener shape"));
                }
            }
        };
    }
}
```

[CITED: archunit.org/userguide/html/000_Index.html ŌĆö `methods().that().areAnnotatedWith(...)` syntax + custom `ArchCondition`] ŌĆö verified for ArchUnit 1.4.x.

**Why this rule shape is sufficient:** the regression in commit `2b61689` was that a consumer used `@RabbitListener` without a `Message` parameter AND the factory was set to `MANUAL` ŌĆö so any future consumer that adds a `Channel` parameter is flagged immediately. The rule does NOT inspect the factory bean (ArchUnit can't), but the contract enforces the LISTENER SIGNATURE and that's the testable invariant.

**Alternative (simpler): plain JUnit + reflection.** If ArchUnit feels heavyweight, an equivalent `~30 LOC` test using `Reflections` library + `Class.forName().getDeclaredMethods()` works ŌĆö but ArchUnit is already a recognized standard, the DSL reads more declaratively, and there are likely additional ArchUnit rules to add in Phase 6+ (e.g., "every saga consumer must have an @Transactional service delegate" ŌĆö already a Phase 4 lesson). Pick ArchUnit.

### Pattern 6: Saga Consumer (Inherited from Phase 4 ŌĆö DO NOT DEVIATE)

**Canonical shape verbatim from `inventory-service.OrderCreatedConsumer` + `InventoryOrderService`:**

```java
@Component
public class StockReservedConsumer {
    private final OrderSagaService orderSagaService;
    private final ObjectMapper objectMapper;

    public StockReservedConsumer(OrderSagaService orderSagaService, ObjectMapper objectMapper) { ŌĆ” }

    @RabbitListener(queues = OrderRabbitConfig.QUEUE_ORDER_STOCK_RESERVED)
    public void handleStockReserved(Message amqpMessage) {
        // 1) deserialize envelope; throw AmqpRejectAndDontRequeueException on bad JSON / null payload / bad UUID ŌåÆ DLQ
        // 2) treeToValue payload ŌåÆ StockReservedPayload record
        // 3) call orderSagaService.processStockReserved(eventId, envelope, payload) ŌĆö DELEGATE TRANSACTION
        // 4) on JsonProcessingException at step 3, rethrow as RuntimeException ŌåÆ StatefulRetryInterceptor retries ŌåÆ DLQ after 3 attempts
    }
}

@Service
public class OrderSagaService {
    @Transactional
    public void processStockReserved(UUID eventId, Envelope envelope, StockReservedPayload payload) {
        if (processedEventsRepository.existsById(eventId)) return;  // idempotent
        Order order = orderRepository.findById(payload.orderId())
            .orElseThrow(() -> new AmqpRejectAndDontRequeueException("Order not found"));
        order.setStatus(OrderStatus.STOCK_RESERVED);
        order.setUpdatedAt(Instant.now());
        orderRepository.save(order);
        // (no outbox event here ŌĆö wait for payment.completed)
        processedEventsRepository.save(new ProcessedEvent(eventId, "StockReservedConsumer", envelope.eventType()));
    }
}
```

**Key invariants** (already in inventory-service code; CLAUDE.md Rule #3):
1. `@RabbitListener` is NOT `@Transactional` (Plan 04-02 lesson ŌĆö AOP proxy can be bypassed on AMQP container thread).
2. The `@Service` delegate IS `@Transactional`. Idempotency check via `processedEventsRepository.existsById(eventId)` happens INSIDE the transaction.
3. `processed_events` row is written in the SAME transaction as the side effect, ALWAYS (success OR failure path ŌĆö failure-path row prevents reprocessing the same poison message).
4. Throws `AmqpRejectAndDontRequeueException` for unrecoverable cases (malformed message, missing parent entity); container nacks-no-requeue ŌåÆ DLQ.
5. Other exceptions propagate; `StatefulRetryInterceptor` retries 3 times (delays 1s, 5s) then DLQ.

### Pattern 7: Order State Machine in `@Transactional`
**What:** Each saga consumer transitions order status atomically; a failed transition (e.g., consumer arrives for a CANCELLED order) is logged and the row is left as-is.

### Anti-Patterns to Avoid

- **Sync HTTP call inside `@Transactional`.** [VERIFIED: marcobehler.com / spring docs] ŌĆö locks DB connection while waiting on remote response. **In Phase 5**: cart read, address read, product price re-validation MUST happen BEFORE `@Transactional` opens. Service shape:
  ```java
  public OrderCreationOutcome createOrder(UUID userId, UUID idempotencyKey, CreateOrderRequest body) {
      // 1. Pre-tx ŌĆö sync REST calls, idempotency check, drift detection
      Cart cart = cartClient.getCart(userId);
      List<ProductSnapshot> products = productClient.getByIds(cart.getProductIds());
      AddressSnapshot address = identityClient.getAddress(body.addressId(), userId);
      validatePriceDrift(cart, products);  // throws 409 PriceDriftException if drifted

      // 2. Tx ŌĆö DB writes only
      return orderTransactionalService.persistOrder(userId, idempotencyKey, body, cart, products, address);
  }
  ```
- **Manual ack mode without `Channel` parameter.** D-10 ArchUnit gate enforces. (Plan 04-03 retrospective.)
- **`@Transactional` on the `@RabbitListener` method itself.** Spring AMQP 3.2.x's container thread bypasses the AOP proxy. Always delegate to a `@Service` (Phase 4 lesson).
- **Forgetting `MessageProperties.setMessageId(eventId)`.** `StatefulRetryInterceptor.messageKeyGenerator` rejects the message with `AmqpException` BEFORE the listener runs. D-09's shared `OutboxMessagePostProcessor` makes this uniform.
- **Hibernate `save()` without UPSERT for cart-add** ŌĆö races under concurrent adds. Always native UPSERT.
- **JSON-stringified amounts in saga payloads.** Plan 04-03 retrospective: `totalAmount` and `unitPrice` MUST be JSON numbers, not strings. Schema validates `{"type": "number", "minimum": 0}`.
- **Cross-schema JOINs in queries.** Schema-per-service is structural; `cart_user` cannot SELECT from `orders.*` etc. Cross-service data goes through REST or saga events ONLY.

## Common Pitfalls

### Pitfall 1: Sync HTTP call inside `@Transactional` boundary
**What goes wrong:** `POST /orders` calls `cartClient.getCart()` from inside an `@Transactional` method; the DB connection is held while the HTTP request blocks; under load Hikari pool exhausts; first 502 follows.
**Why it happens:** Java services tend to nest "business operation = single transaction"; HTTP looks like business logic.
**How to avoid:** Two-step shape ŌĆö pre-tx fetches, then `orderTransactionalService.persistOrder(...)` runs in `@Transactional`. (See "Anti-Patterns" Pattern 7 example.)
**Warning signs:** Hikari pool warnings in JSON logs (`HikariCP - Connection is not available`), 30s+ p99 on `POST /orders`, Postgres `pg_stat_activity` rows stuck in `idle in transaction` state for seconds.

### Pitfall 2: Idempotency-Key race on first call
**What goes wrong:** Two simultaneous requests with same `(idempotency_key, user_id)` arrive; both pass step 1's existence check; both try to INSERT into `order_idempotency_keys` at commit; one throws `DataIntegrityViolationException`.
**Why it happens:** Stripe's pattern requires read-then-write; the gap is racy.
**How to avoid:** Treat the constraint violation as the loser-path ŌĆö catch `DataIntegrityViolationException`, re-run step 1 to fetch the winner's `order_id`, return its response. Brandur Leach's article calls this "let the database be the lock."
**Warning signs:** `DataIntegrityViolationException: duplicate key value violates unique constraint "order_idempotency_keys_pkey"` in logs without follow-up SELECT; client sees 500 instead of 200.

### Pitfall 3: Outbox poller publishing without `messageId` set
**What goes wrong:** `RabbitTemplate.convertAndSend(...)` publishes a message without `messageId` property; consumer's `StatefulRetryInterceptor.messageKeyGenerator` throws `AmqpException`; message dropped before listener runs; saga stalls silently.
**Why it happens:** Plan 04-03 retrospective ŌĆö already happened once in this codebase (commit `06338b1` was the smoke-runbook fix). D-09's `OutboxMessagePostProcessor` is the structural fix.
**How to avoid:** ALL outbox publishers go through `AbstractOutboxPoller.poll()`, which calls `convertAndSend(exchange, routingKey, payload, outboxMessagePostProcessor())`. The post-processor extracts `eventId` from the envelope JSON and sets `setMessageId(eventId)`.
**Warning signs:** Consumer's `processed_events` count stays at 0 after a publish; rabbitmq management UI shows the queue draining but with `Get Message` tab showing the message has no `message_id` property; `AmqpException("Stateful retry requires a non-null messageId")` in service logs.

### Pitfall 4: Cross-schema JOIN attempt
**What goes wrong:** order-service tries `SELECT ŌĆ” FROM orders.orders o JOIN cart.cart_items c ON c.user_id = o.user_id`; Postgres returns `permission denied for schema cart` because `orders_user` was REVOKEd USAGE on `cart` schema.
**Why it happens:** Convenience ŌĆö schemas are on the same Postgres host, so a developer reaches for a JOIN.
**How to avoid:** order-service NEVER queries `cart.*` directly. It calls `GET /api/v1/cart` (gateway-routed; cart-service serves) and snapshots into `orders.order_items`.
**Warning signs:** `org.postgresql.util.PSQLException: ERROR: permission denied for schema cart`; explicit smoke test `infra-tests/CrossSchemaDenyTest` (Plan 01-08) is supposed to fail on any such regression.

### Pitfall 5: Listener method missing `Message` parameter ŌĆö silent ack-mode regression
**What goes wrong:** A future Phase 6+ consumer is copy-pasted from a non-saga template that uses `@RabbitListener` with a deserialized POJO parameter (`public void handle(OrderCreated event)`); the factory's MANUAL ack mode silently nacks every successful message; infinite redelivery on next reconnect.
**Why it happens:** Plan 04-03 already happened ŌĆö the bug's vector is a copy-paste of an outdated example.
**How to avoid:** D-10 ArchUnit gate fails the build immediately on any non-canonical listener shape.
**Warning signs:** ArchUnit test fails with a violation message naming the specific class.

### Pitfall 6: Awaitility test races with Hibernate first-level cache
**What goes wrong:** Test calls `consumer.handleStockReserved(...)`; immediately calls `orderRepository.findById(orderId).get().getStatus()` and asserts `STOCK_RESERVED`. The change wrote in a different transaction ŌĆö the test's session has the old entity cached.
**Why it happens:** Spring Data JPA's `findById` returns the first-level cached entity if present.
**How to avoid:** In the test, use `JdbcTemplate` for assertions (Phase 4 pattern in `OrderCreatedConsumerIntegrationTest` ŌĆö see line 230-247) OR call `entityManager.clear()` before re-fetching. The Phase 4 pattern is cleaner.
**Warning signs:** Test passes when run alone, fails when run in a suite; assertions intermittently see "the previous value".

### Pitfall 7: Cart-clear consumer fires before order-confirmed is durable
**What goes wrong:** order-service publishes `order.confirmed` from outbox poller; cart-service's `cart.q.order-confirmed` consumer receives it; cart-service's `processed_events` insert succeeds but the cart_items DELETE fails ŌĆö RabbitMQ delivers again ŌĆö but processed_events already has the row, so the cart is never cleared.
**Why it happens:** Idempotency check OUTSIDE the transaction.
**How to avoid:** EVERY consumer's `@Transactional` writes processed_events INSIDE the same tx as the side effect (the lifted-from-Phase-4 pattern). If the DELETE fails, the processed_events row rolls back too; redelivery retries. (CLAUDE.md Rule #3.)
**Warning signs:** Cart shows old items even after `Sipari┼¤lerim` shows the order as CONFIRMED.

### Pitfall 8: Order created with stale price, frontend re-renders, user sees different total
**What goes wrong:** User adds item at 100 TRY; price drops to 80 TRY between cart-add and `POST /orders`; frontend shows 80 TRY but the snapshot in `cart_items` still says 100; the price-drift check at order-create returns 409 with `updatedItems`, but if the frontend doesn't handle 409 gracefully the user sees a generic error.
**Why it happens:** D-01 deliberately allows this ŌĆö the snapshot is canonical, the user agreed to 100 TRY.
**How to avoid:** Frontend (Phase 10) MUST handle the 409 by re-rendering the cart with the new prices and asking "fiyatlar g├╝ncellendi, devam etmek ister misin?" before retrying.
**Warning signs:** Backend log shows `OrderPriceDriftException` followed by no retry; user-side console shows generic 500.

### Pitfall 9: Mock payment-service publishes `payment.completed` BEFORE order-service has set `STOCK_RESERVED`
**What goes wrong:** Saga timing edge ŌĆö `stock.reserved` event fans out to BOTH `order.q.stock-reserved` AND `payment.q.stock-reserved`; payment-service is faster (only mock 100ms delay); publishes `payment.completed`; order-service's `payment-completed` consumer sees the order still in `PENDING` state.
**Why it happens:** Choreography saga has no global ordering.
**How to avoid:** Order state machine accepts `payment.completed` from BOTH `PENDING` and `STOCK_RESERVED` states (transition to `PAID` either way). Document as an intentional design choice in the state-transition table. The `processed_events` inbox prevents duplicate processing.
**Warning signs:** Awaitility test flakes ŌĆö passes 80% of the time.

### Outbox publish from `@Transactional` service

```java
// OrderCreationService ŌĆö pre-tx fetches done; this method just persists
@Service
public class OrderTransactionalService {

    @Transactional
    public OrderCreationOutcome persistOrder(
            UUID userId, UUID idempotencyKey, CreateOrderRequest body,
            Cart cart, List<ProductSnapshot> products, AddressSnapshot address) {

        // 1) Idempotency-Key dedup ŌĆö try insert
        Optional<OrderIdempotencyKey> existing = idempotencyKeyRepo
            .findByIdempotencyKeyAndUserId(idempotencyKey, userId);
        if (existing.isPresent()) {
            Order replayed = orderRepository.findById(existing.get().getOrderId()).orElseThrow();
            return OrderCreationOutcome.replay(replayed);
        }

        // 2) Persist domain rows
        UUID orderId = UUID.randomUUID();
        UUID correlationId = orderId;  // saga-initiating event: correlationId = orderId

        Order order = new Order(orderId, userId, OrderStatus.PENDING,
                computeTotal(cart, products), "TRY", correlationId, idempotencyKey);
        orderRepository.save(order);

        for (CartLine line : cart.lines()) {
            orderItemRepository.save(new OrderItem(orderId, line.productId(),
                line.nameSnapshot(), line.qty(), line.unitPriceSnapshot()));
        }

        orderShippingAddressRepository.save(new OrderShippingAddress(orderId, address));

        // 3) Idempotency-Key row
        idempotencyKeyRepo.save(new OrderIdempotencyKey(idempotencyKey, userId, orderId));

        // 4) Outbox: order.created envelope
        UUID eventId = UUID.randomUUID();
        OrderCreatedPayload payload = new OrderCreatedPayload(
            orderId, userId, "TRY", order.getTotalAmount(),
            cart.lines().stream()
                .map(l -> new OrderItemPayload(l.productId(), l.qty(),
                    l.unitPriceSnapshot(), l.nameSnapshot()))
                .toList()
        );
        Envelope envelope = new Envelope(
            eventId.toString(), "order.created", 1, Instant.now(),
            correlationId.toString(),  // saga root: correlationId = own eventId... wait ŌĆö saga-contracts.md ┬¦1 says correlationId = orderId. We use orderId here.
            null,                       // saga root has null causation
            "order-service",
            objectMapper.valueToTree(payload)
        );
        String envelopeJson;
        try {
            envelopeJson = objectMapper.writeValueAsString(envelope);
        } catch (JsonProcessingException e) {
            throw new IllegalStateException("Failed to serialize order.created envelope", e);
        }
        orderOutboxRepository.save(new OutboxEvent(
            eventId, "orders", "order.created", envelopeJson, Instant.now()
        ));

        return OrderCreationOutcome.created(order);
    }
}
```

```java
@RestController
@RequestMapping("/cart")
public class CartController {

    private static final String HEADER_USER_ID = "X-User-Id";

    private final CartService cartService;

    public CartController(CartService cartService) { this.cartService = cartService; }

    @GetMapping
    public CartView get(HttpServletRequest req) {
        UUID userId = resolveUserId(req);
        return cartService.getCart(userId);  // lazy-creates empty cart per D-11
    }

    @PostMapping("/items")
    @ResponseStatus(HttpStatus.CREATED)
    public CartView add(HttpServletRequest req, @Valid @RequestBody AddCartItemRequest body) {
        UUID userId = resolveUserId(req);
        return cartService.addItem(userId, body.productId(), body.qty());  // upsert + sync product fetch
    }

    @PatchMapping("/items/{productId}")
    public CartView update(HttpServletRequest req,
                            @PathVariable UUID productId,
                            @Valid @RequestBody UpdateCartItemRequest body) {
        UUID userId = resolveUserId(req);
        return cartService.updateQty(userId, productId, body.qty());
    }

    @DeleteMapping("/items/{productId}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void remove(HttpServletRequest req, @PathVariable UUID productId) {
        UUID userId = resolveUserId(req);
        cartService.removeItem(userId, productId);
    }

    private UUID resolveUserId(HttpServletRequest req) {
        String h = req.getHeader(HEADER_USER_ID);
        if (h == null || h.isBlank()) throw new ResponseStatusException(HttpStatus.UNAUTHORIZED, "Kimlik do─¤rulamas─▒ gerekli");

        catch (IllegalArgumentException e) { throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Ge├¦ersiz kullan─▒c─▒ kimli─¤i"); }
    }
}
```
