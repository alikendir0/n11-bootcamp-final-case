  <files>payment-service/src/main/java/com/n11/payment/api/PaymentController.java, payment-service/src/main/java/com/n11/payment/api/IyzicoCallbackRequest.java, payment-service/src/main/java/com/n11/payment/messaging/PaymentTransactionalService.java, payment-service/src/test/java/com/n11/payment/api/IyzicoCallbackIntegrationTest.java</files>

  <behavior>
    - Test 1: successful retrieve updates payment to COMPLETED and writes exactly one `payment.completed` outbox event.
    - Test 2: failed retrieve updates payment to FAILED and writes exactly one `payment.failed` outbox event.
    - Test 3: duplicate callback for the same token is idempotent and writes no second outbox event.
  </behavior>
  <action>Add `POST /payments/iyzico/callback` consuming **`application/x-www-form-urlencoded`** with field `token` (this is how Iyzico Checkout Form posts the user's browser back to the merchant — confirmed against Iyzico docs and `iyzipay-java` sample). Bind via `@RequestParam("token") String token`, not `@RequestBody`. Do not accept JSON for this endpoint; the S2S webhook (separate JSON feature) is out of scope for Phase 6.

  Additionally handle the mdStatus edge cases surfaced by sandbox cards `4131 1111 1111 1117` (mdStatus=0) and `4141 1111 1111 1115` (mdStatus=4): treat any retrieve response with `paymentStatus != "SUCCESS"` as `IYZICO_DECLINED` regardless of HTTP-level success.

  On callback, find payment by `iyzicoToken`; call `IyzicoCheckoutClient.retrieve(token, orderId/correlation conversation id)`; trust neither callback body nor status until retrieve succeeds and adapter verifies signature via `CheckoutForm.verifySignature(secretKey)`. In `PaymentTransactionalService`, add terminal methods guarded by current status `PENDING` only: `completeFromCallback` writes `payment.completed` payload `(orderId, paymentId, iyzicoPaymentId, amount, currency)`; `failFromCallback` writes `payment.failed` payload `(orderId, paymentId, reason, errorCode)`. Reasons should distinguish `IYZICO_DECLINED`, `IYZICO_FRAUD_REVIEW`, `IYZICO_RETRIEVE_FAILED`, `IYZICO_3DS_MDSTATUS_INVALID`. Duplicate terminal callbacks return the current terminal status without extra outbox rows. Return HTTP 200 with a minimal `text/html` result page (or 302 redirect to a success/failure URL) — Iyzico's hosted page renders the response inside the user's browser, so empty/JSON responses degrade UX.</action>
  <verify><automated>./gradlew :payment-service:test --tests '*IyzicoCallbackIntegrationTest'</automated></verify>
  <acceptance_criteria>`IyzicoCallbackIntegrationTest` asserts one outbox row for success and one for failure; `PaymentController.java` contains `iyzico/callback`, `consumes = MediaType.APPLICATION_FORM_URLENCODED_VALUE`, and `@RequestParam("token")`; callback code calls `retrieve` and `verifySignature`; mdStatus-invalid path emits `payment.failed` with reason `IYZICO_3DS_MDSTATUS_INVALID`.</acceptance_criteria>
  <done>Iyzico callback drives the saga via payment.completed/payment.failed with idempotent terminal transitions.</done>
