<objective>
Implement QUAL-04 — the saga happy-path integration test that closes the loop. Per ROADMAP.md SC-3: "The saga happy-path integration test (QUAL-04) covers `OrderCreated → StockReserved → PaymentCompleted → OrderConfirmed → notification logged`, using Testcontainers + Awaitility, and passes on green CI."

Two-tier scope per RESEARCH.md §Planning Inputs item 3:
- (a) Per-consumer idempotency tests — already covered by Plan 07-04
- (b) **THIS plan**: a narrow E2E test in infra-tests that publishes `order.confirmed` directly and asserts the notification row appears in the `notifications` table within ~10s. This is the architecturally sufficient QUAL-04 deliverable: it proves live AMQP delivery → consumer invocation → @Transactional persist → DB row, all running on real Testcontainers RabbitMQ + Postgres.

The test does NOT need to chain `order.created → stock.reserved → payment.completed → order.confirmed` (the existing `SagaHappyPathE2ETest` already proves the pre-notification chain end-to-end). QualFourSagaNotificationTest extends that proof to include the `→ notification logged` final hop. Together, both tests cover SC-3 with overlapping witness.

Purpose: Without this test, the saga loop has no automated proof of closure — only the per-consumer idempotency tests run. QUAL-04 is the contract that survives every regression: any future change to NotificationRabbitConfig, OrderConfirmedConsumer, or NotificationService will fail this test if it breaks the AMQP wiring or the DB persist path. The test is also a structural confirmation that infra-tests' multi-service classpath disambiguation pattern (Plan 05-04 STATE.md) extends cleanly to a fourth service.
