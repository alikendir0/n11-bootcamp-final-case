Per CD-02 the archetype is a **hybrid**: it ships as a real runnable Gradle subproject (so the root build catches archetype drift early — if the template breaks, the entire build fails before any service inherits the break) AND as a `service-template/skeleton/` copy-paste directory tree with placeholder tokens (so each business service starts from a known-good baseline, then owns its divergent copy independently). The archetype wires the four D-11 items end-to-end:

1. **Boot 3.5.14 + Eureka client + Springdoc + Actuator** — runnable Boot app that registers in Eureka and exposes `/actuator/health`, `/actuator/info`, `/v3/api-docs`, `/swagger-ui.html`, `/sample` (the smoke endpoint that proves the wiring works).

4. **Flyway 12.5 + `flyway-database-postgresql:12.5.0` non-owner-user wiring** — `spring.flyway.create-schemas: false`, `default-schema: ${flyway.schema}`, `schemas: ${flyway.schema}`. ## CD-02 Hybrid Decision Rationale (locked here per CONTEXT.md `<specifics>`)

The user's CONTEXT.md asked the planner to flag the trade-off between:
- **Pure skeleton:** `service-template/` is just a copy-paste directory; nothing builds at root; nothing catches archetype drift until a service tries to clone it.
- **Pure subproject:** `service-template/` is a real Gradle subproject; later services depend on it via `apply from:` or shared classpath; introduces tight coupling that makes per-service drift painful.

**Pick: hybrid.** The archetype is BOTH a real runnable subproject (`./gradlew :service-template:build` runs in the root build, catching drift early) AND ships a `service-template/skeleton/` directory with `<SERVICE_NAME>` / `__SERVICE_PACKAGE__` placeholder tokens that Phase 3+ copies into a new sibling subproject (identity-service/, product-service/, …). This gives:

1. **Drift catch via subproject:** if the archetype's wiring breaks (e.g., a Flyway upgrade flips a default), `./gradlew build` fails at the archetype before any business service inherits the break.
2. **Independence via skeleton:** business services don't depend on the archetype subproject at runtime. They copy the skeleton tree, rename placeholders, and own divergent copies. A change in service-template doesn't ripple into existing services automatically — it only affects the next service that's scaffolded.
3. **Single source of truth for the archetype shape:** changes go in `service-template/skeleton/`, then re-copy into the runnable subproject (or vice-versa, as long as both stay aligned). Document the procedure in `skeleton/README.md`.

**Trade-off acknowledged:** the hybrid means two locations for the same code (the runnable subproject under `service-template/src/main` AND the skeleton under `service-template/skeleton/src-main`). To prevent drift between THESE two, Task 7 includes a verification step that diffs the two trees (modulo placeholder substitution) and the README documents that any change must be applied to both. An alternative would be to have the runnable subproject programmatically copy from the skeleton at build time — that's a Phase-11 cleanup if drift becomes a problem; for Phase 1, manual diff is sufficient.
</objective>

**Production posture for `optional:` + `fail-fast=true` (revised — addresses checker WARNING):**

These two appear contradictory but live at different layers:
- `optional:` is a Spring Boot config-import resolver prefix: "if the URI scheme handler does not respond at all, log and continue."
- `fail-fast=true` is a config-server CLIENT retry parameter: "if I DO connect and exhaust the retry budget, abort the boot."

The original draft kept `optional:` for IDE/test convenience, but in a production-shape compose path this weakens ARCH-11 cold-boot resilience — a misnetworked config-server could let services silently boot with NO externalized config at all. ARCH-11 wants the boot to fail in that case so the operator sees the misconfiguration.

**Resolution for Phase 1 archetype:** keep `optional:` in this archetype's `application.yml` because the `service-template` subproject is BUILT and tested by `./gradlew :service-template:build` in CI (no config-server running) — without `optional:` the build itself would fail. Each business service (Phase 3+) cloning from `service-template/skeleton/` STRIPS `optional:` from its production `application.yml` so its production posture is "config-server is required; if absent, fail loud." The skeleton's `README.md` (Task 5) documents this as a clone-time edit.

This trade-off is recorded in must_haves.truths and in `01-07-SUMMARY.md`. The acceptance criteria for this plan verifies `optional:` is present in `service-template/src/main/resources/application.yml` (build-safety) AND that `service-template/skeleton/README.md` documents the strip-on-clone instruction.

1. `cp -r service-template/skeleton/src-main <new-service>/src/main`
2. Rename the placeholder package directory `__SERVICE_PACKAGE__` to the new service's package leaf
3. Drop the `.template` suffix from every file
4. `find <new-service>/src/main -type f -exec sed -i 's/<SERVICE_NAME>/<new-service-name>/g; s/__SERVICE_PACKAGE__/<new-service-package>/g' {} \;`
5. Author a new `<new-service>/build.gradle.kts` (copy from `service-template/build.gradle.kts` and adjust dependencies for the service's actual needs)

### File 1: `service-template/skeleton/README.md`

```markdown
# service-template/skeleton/

Copy-paste source tree for scaffolding a new business service. Used by Phase 3+ executors.

## Why a `.template`-suffixed mirror instead of plain source?

- The `.template` suffix prevents the skeleton files from being picked up by the runnable
  `service-template` subproject's compile classpath (the runnable subproject reads from
  `service-template/src/main/`, NOT `service-template/skeleton/src-main/`).
- The runnable subproject (under `src/main/`) and the skeleton tree are kept in sync MANUALLY.
  Any change to one must be applied to the other; Task 7's verification step catches drift.
- See PLAN.md `<objective>` "CD-02 Hybrid Decision Rationale" for the design trade-off.

## Placeholder tokens

| Token | Meaning | Example replacement |
|-------|---------|---------------------|
| `<SERVICE_NAME>` | Spring application name (kebab-case) | `identity-service`, `product-service`, `cart-service` |
| `__SERVICE_PACKAGE__` | Java package leaf (lowercase, no hyphens) | `identity`, `product`, `cart` |

## Clone procedure (Phase 3+ executors run these)

```bash
SERVICE_NAME=identity-service          # kebab-case Spring app name
SERVICE_PACKAGE=identity               # Java package leaf

# 1. Copy skeleton tree into a new sibling subproject directory.
mkdir -p ${SERVICE_NAME}/src
cp -r service-template/skeleton/src-main ${SERVICE_NAME}/src/main

# 2. Rename the placeholder package directory.
mv ${SERVICE_NAME}/src/main/java/com/n11/__SERVICE_PACKAGE__ \
   ${SERVICE_NAME}/src/main/java/com/n11/${SERVICE_PACKAGE}

# 3. Drop the .template suffix from every file.
find ${SERVICE_NAME}/src/main -type f -name '*.template' | while read f; do
  mv "$f" "${f%.template}"
done

# 4. Substitute placeholder tokens in file contents.
find ${SERVICE_NAME}/src/main -type f \( -name '*.java' -o -name '*.yml' -o -name '*.xml' -o -name '*.sql' \) -exec \
  sed -i \
    -e "s/<SERVICE_NAME>/${SERVICE_NAME}/g" \
    -e "s/__SERVICE_PACKAGE__/${SERVICE_PACKAGE}/g" \
  {} \;

# 5. Add the new subproject to settings.gradle.kts (root):
#    include("${SERVICE_NAME}")

# 6. Author ${SERVICE_NAME}/build.gradle.kts by copying service-template/build.gradle.kts and
#    pruning/adding dependencies as needed for this service's domain.

# 7. Author config-server/src/main/resources/config/${SERVICE_NAME}.yml by copying
#    service-template.yml and overriding:
#       - server.port: <unique port>
#       - flyway.schema: ${SERVICE_PACKAGE}        (or the matching schema name from 01-03 init.sh)
#       - spring.datasource.username/password: env-var refs to ${SERVICE_PACKAGE}_user
#    (See 01-03's init.sh for the canonical schema/user mapping.)

# 8. Author ${SERVICE_NAME}/src/main/resources/db/migration/V2__<feature>.sql for the service's
#    domain entities. V1__init_processed_events.sql comes from the skeleton untouched.
```

Any change to `service-template/src/main/...` MUST be re-applied to `service-template/skeleton/src-main/...`
