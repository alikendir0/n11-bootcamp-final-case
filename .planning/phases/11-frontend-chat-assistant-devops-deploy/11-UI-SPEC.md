## Spacing Scale

Use the existing Phase 10 Tailwind 4 `@theme` spacing tokens from `frontend/src/index.css`; do not introduce new spacing values.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, status-chip inner gaps, transcript timestamp gaps |
| sm | 8px | Message bubble padding increments, compact product-card gaps |
| md | 16px | Default drawer padding, transcript message gap, input row gap |
| lg | 24px | Drawer header/footer padding, compact card padding, section spacing inside docs/demo cards |
| xl | 32px | Drawer-to-edge offset on desktop where applicable, empty-state vertical rhythm |
| 2xl | 48px | Full-screen mobile sheet section spacing |
| 3xl | 64px | Not used inside chat; reserved for page-level surfaces already defined by Phase 10 |

Exceptions:
- Floating chat bubble: **56px × 56px** fixed circular touch target. This is intentionally larger than the 44px accessibility minimum and aligns with common chat-widget affordance size.
- Drawer desktop width: **420px** exactly from `11-CONTEXT.md` D-02.
- Mobile sheet: **100vw × 100dvh**, no side gutter; internal padding still uses the scale above.
- Chat transcript should keep a minimum **64px bottom padding** above the composer so the final streamed token is never hidden behind the input row.

| Role | Token / Hex | Usage |
|------|-------------|-------|
| Dominant (60%) | `--color-body-bg` `#EDEFF3` | Page backdrop behind drawer overlay, transcript background on desktop |

- Border: `--color-border` `#E5E7EB` for drawer seams, card borders, composer border, chip outlines.

- Assistant message bubble: white background + black text + `#E5E7EB` border.
- Tool status chip: white/transparent background + black text + border; success adds green dot; failure adds red dot.

## Chat Assistant Visual and Interaction Contract

### Placement and Shell

- Mount the chat widget in `frontend/src/components/layout/Layout.tsx` after `AuthEventBridge`/toast bridges so it persists across all routed pages.
- Floating trigger: bottom-right, `right: 24px`, `bottom: 24px` on desktop; `right: 16px`, `bottom: 16px` on mobile. Size 56×56, circular, black background, white lucide icon, `aria-label="Yapay Zeka Alışveriş Asistanını Aç"`.
- Desktop open state: right-side drawer, fixed, top 0, right 0, height 100dvh, width 420px, white surface, left border `#E5E7EB`, shadow. Use an overlay scrim only at `rgba(0,0,0,0.16)` so PDP/cart context remains visible during the demo.
- Mobile open state: full-screen sheet, 100vw × 100dvh, own header and close button. Do not render a narrow drawer below 768px.
- Drawer header: title `Yapay Zeka Alışveriş Asistanı`; subtitle `Ürün bulma, sepet ve sipariş işlemlerinde yardımcı olur.`; close button with `aria-label="Asistanı Kapat"`.
- State persists across route navigation. Closing hides the panel but does not clear conversation state.

### Transcript and Streaming

- Transcript is a vertical scroll region with `role="log"`, `aria-live="polite"`, and `aria-relevant="additions text"`.
- User messages align right with black bubble; assistant messages align left with white bordered bubble.
- Stream Gemini `delta` events token-by-token into the active assistant bubble without blocking composer typing or page scrolling.
- During active streaming, show a small label under the assistant bubble: `Yanıt yazılıyor...` at Label size. Remove it on `done`.
- Auto-scroll only if the user is already within 80px of the transcript bottom; if the user has scrolled upward, show a compact `Yeni mesajlar` jump button.
- Composer remains enabled during normal idle; during active stream, disable send and show `Gönderiliyor...`. Mid-stream cancellation is v2 and must not be added unless trivial.

### Tool-Use Chips

Map typed SSE events to concise Turkish chips in transcript order:

| Event / Tool | Chip Copy |
|--------------|-----------|
| `search_products` `tool_call` | `Ürünler aranıyor...` |
| `get_product` `tool_call` | `Ürün detayı getiriliyor...` |
| `list_categories` `tool_call` | `Kategoriler hazırlanıyor...` |
| `add_to_cart` `tool_call` | `Sepete ekleniyor...` |
| `update_cart_item` / `remove_from_cart` `tool_call` | `Sepet güncelleniyor...` |
| `view_cart` `tool_call` | `Sepet kontrol ediliyor...` |
| `create_order` `tool_call` | `Sipariş hazırlanıyor...` |
| `get_payment_link` `tool_call` | `Ödeme bağlantısı hazırlanıyor...` |
| `get_order_status` `tool_call` | `Sipariş durumu kontrol ediliyor...` |
| `tool_result.ok === true` | `{summary}` if Turkish and concise; otherwise `İşlem tamamlandı.` |
| `tool_result.ok === false` | `İşlem tamamlanamadı.` + inline retry/auth guidance if applicable |

Chip visual: Label 14px, border `#E5E7EB`, radius full, max one line with ellipsis. Pending chip uses spinner; success chip uses green dot; failure chip uses red dot.

### Compact Cards in Chat

- Product result card: max width 100%, white background, border, 8px radius. Content: title (2-line clamp), `formatTRY(priceGross)`, stock/free-shipping hint when available, actions `Ürünü Gör` and `Sepete Ekle`.
- Product cards must stay compact: no gallery, no 4-column grid, no PDP tabs inside chat.
- `Ürünü Gör` navigates to existing `/urun/:slug-:id` route and keeps the drawer open.
- `Sepete Ekle` uses the same cart mutation path as the storefront. After confirmed success, invalidate `cartQueryKey = ['cart']`; header cart badge must update within 1 second; show toast `Ürün sepete eklendi.` Do not auto-open `/sepetim`.
- Cart summary card: show item count, subtotal/total using `formatTRY`, and CTA `Sepete Git` -> `/sepetim`.
- Order/payment card: show order status in Turkish and CTAs `Ödemeye Git` or `Ödemeye Devam Et`; never auto-navigate to checkout or Iyzico.

### Authentication and Error States

- Guest mutating-tool auth-required state: assistant message `Sepete eklemek ve sipariş işlemleri için giriş yapmanız gerekiyor.` + CTA `Giriş Yap` -> `/giris-yap?redirectUrl=<currentPath>`. Preserve current page in `redirectUrl`.
- SSE/Gemini failure state: inline transcript error card with copy `Yanıt alınamadı. Lütfen tekrar deneyiniz.` + CTA `Tekrar dene`, which re-sends the same last user message with the same `conversationId`.
- Empty chat state: heading `Merhaba, size nasıl yardımcı olabilirim?`; body `Ürün arayabilir, sepetinizi düzenleyebilir veya sipariş durumunuzu öğrenebilirsiniz.`; suggested prompts: `MacBook ara`, `Sepetimi göster`, `Sipariş durumumu kontrol et`.
- Conversation ID: frontend-generated UUID persisted in browser storage. Do not expose raw UUID in visible UI.
- All chat network calls use `${VITE_API_BASE_URL}/api/v1/chat/stream` through the gateway only; no direct `ai-service` URL.

## DevOps / Demo Documentation UX Contract

Phase 11 also owns README/runbook surfaces. They are documentation, but they must be scan-friendly during an interview.

- Root `README.md` starts with a one-command demo path: build/pull images, `docker compose --profile full up`, tunnel check, external curl proof.
- Use Turkish product-demo labels where referencing the storefront, but DevOps commands and commit/workflow identifiers remain English.
- Slack examples in docs use exact copy: `✅ build green on <ref>` and `❌ build failed on <ref>`.
- Environment matrix must visually separate required secrets (`GEMINI_API_KEY`, `IYZICO_*`, `JWT_PRIVATE_KEY`, `SLACK_WEBHOOK_URL`, `CLOUDFLARE_TUNNEL_TOKEN`/`NGROK_AUTHTOKEN`) from non-secret public config (`VITE_API_BASE_URL`, `PUBLIC_BASE_URL`).
- Never render real secret examples; use `<set-in-env>` placeholders only.
